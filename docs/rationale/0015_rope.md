# Rope -- Math Rationale

## Overview

The rope demo simulates a chain of nodes hanging under gravity, connected by
distance constraints solved via iterative relaxation. It uses Verlet integration
for position-based dynamics with implicit velocity, rendered via Bevy Gizmos.
The simulation runs on a fixed timestep at 120 Hz for deterministic behavior.

## 1. Verlet Integration

```rust
fn verlet_integrate(mut rope: ResMut<Rope>, time: Res<Time>) {
    let dt = time.delta_secs();
    for i in 0..NODE_COUNT {
        if rope.pinned[i] {
            continue;
        }
        let current = rope.positions[i];
        let old = rope.old_positions[i];
        let velocity = (current - old) * DAMPING;
        rope.old_positions[i] = current;
        rope.positions[i] = current + velocity + GRAVITY * dt * dt;
    }
}
```

### The Störmer-Verlet method

Standard Verlet integration computes the next position from the current and
previous positions without explicitly storing velocity:

```
x(t+dt) = 2*x(t) - x(t-dt) + a*dt^2
```

This is algebraically equivalent to the code's formulation:

```
velocity = x(t) - x(t-dt)                    // implicit velocity
x(t+dt)  = x(t) + velocity + a * dt^2        // advance
```

The implicit velocity is computed as the difference between the current and
previous positions. No velocity vector is stored -- it is derived on the fly.

### Why Verlet over Euler?

1. **Second-order accuracy**: Verlet has `O(dt^2)` local error vs Euler's
   `O(dt)`, meaning it needs fewer timesteps for the same accuracy
2. **Symplectic**: preserves energy better over long simulations
3. **Constraint-friendly**: position-based constraints (see section 3) work
   naturally because Verlet uses positions as the primary state. Moving a
   position automatically adjusts the implicit velocity for the next frame.
4. **Time-reversible**: `x(t-dt) = 2*x(t) - x(t+dt) + a*dt^2`

### Damping

```rust
let velocity = (current - old) * DAMPING;
```

`DAMPING = 0.99` multiplies the implicit velocity by a factor less than 1 each
frame. This is **velocity damping** (or drag):

```
v_damped = v * 0.99
```

Per frame at 120 Hz, 1% of velocity is lost. Over one second (120 frames):

```
velocity_after_1s = velocity_initial * 0.99^120 ≈ 0.30 * velocity_initial
```

So ~70% of velocity is dissipated per second, causing the rope to settle
quickly after disturbance. Without damping, the rope would oscillate
indefinitely.

### Gravity term

```rust
rope.positions[i] = current + velocity + GRAVITY * dt * dt;
```

`GRAVITY = Vec2::new(0.0, -980.0)` in pixels/second^2. The `dt^2` factor
comes from the Verlet equation: `a * dt^2` is the position change due to
constant acceleration over one timestep.

At 120 Hz: `dt = 1/120 ≈ 0.00833`, so `dt^2 ≈ 0.0000694`. The gravity
contribution per frame is:

```
980 * 0.0000694 ≈ 0.068 pixels/frame
```

This is small per frame but accumulates through the implicit velocity.

## 2. Pinned Nodes

```rust
if rope.pinned[i] {
    continue;
}
```

Pinned nodes skip the integration step entirely. Their positions are only
modified by the mouse drag system, not by physics. The `pinned` vector uses
a `Vec<bool>` where `pinned[0] = true` (the anchor) and all others are `false`.

### Why `Vec<bool>` instead of a component?

The rope is a single `Resource`, not a collection of entities. Using a parallel
boolean array is the simplest approach for a contiguous data structure. Each
node is an index, not an entity, so ECS component-based filtering doesn't apply.

## 3. Distance Constraint Solving

```rust
fn apply_constraints(mut rope: ResMut<Rope>) {
    for _ in 0..CONSTRAINT_ITERATIONS {
        for i in 0..NODE_COUNT - 1 {
            let a = rope.positions[i];
            let b = rope.positions[i + 1];
            let delta = b - a;
            let distance = delta.length();
            if distance < f32::EPSILON {
                continue;
            }
            let diff = (distance - SEGMENT_LENGTH) / distance;
            let correction = delta * 0.5 * diff;

            if !rope.pinned[i] {
                rope.positions[i] += correction;
            }
            if !rope.pinned[i + 1] {
                rope.positions[i + 1] -= correction;
            }
        }
    }
}
```

### The constraint

Each pair of adjacent nodes must be exactly `SEGMENT_LENGTH = 12.0` pixels
apart. After Verlet integration, gravity and inertia will have violated this
constraint. The constraint solver restores it.

### Correction derivation

Given nodes A and B with current distance `d` and desired distance `L`:

```
error = d - L
```

The correction pushes A and B along the axis connecting them to reduce the
error. Each node is moved by half the error (equal mass assumption):

```
direction = (B - A) / d              // unit vector from A to B
correction_magnitude = (d - L) / 2
correction = direction * correction_magnitude
           = (B - A) / d * (d - L) / 2
           = (B - A) * (d - L) / (2 * d)
```

The code computes this as:

```rust
let diff = (distance - SEGMENT_LENGTH) / distance;   // (d - L) / d
let correction = delta * 0.5 * diff;                  // (B - A) * 0.5 * (d-L)/d
```

A gets `+correction` (pushed toward B) and B gets `-correction` (pushed toward
A), splitting the error equally.

### Why 0.5?

The 0.5 factor splits the correction between both nodes (equal mass). If one
node is pinned, the other node receives the full correction because only the
unpinned node's position is modified. This effectively makes the pinned node
infinitely heavy -- it absorbs none of the error.

### The epsilon guard

```rust
if distance < f32::EPSILON {
    continue;
}
```

When two nodes are coincident (`d ≈ 0`), the direction vector is undefined
(division by zero). Skipping degenerate cases prevents NaN propagation. In
practice this rarely happens because gravity quickly separates nodes.

## 4. Iterative Relaxation

```rust
const CONSTRAINT_ITERATIONS: usize = 15;

for _ in 0..CONSTRAINT_ITERATIONS {
    for i in 0..NODE_COUNT - 1 {
        // solve each constraint
    }
}
```

### Why iterate?

Satisfying one constraint may violate adjacent ones. Consider three nodes A-B-C:
- Fixing A-B distance moves B
- B moving violates the B-C constraint
- Fixing B-C moves C, etc.

A single pass through all constraints leaves residual errors. Multiple passes
converge toward a globally consistent solution through **Gauss-Seidel
relaxation**: each constraint uses the latest positions (including corrections
from earlier in the same iteration), propagating fixes through the chain.

### Convergence

For a chain of N nodes, the "stiffness" of the rope depends on iterations:
- **Few iterations** (1-3): rope appears stretchy, like a rubber band
- **Many iterations** (15+): rope appears rigid, like a chain
- **Infinite iterations**: perfect constraint satisfaction

With `NODE_COUNT = 40` and `CONSTRAINT_ITERATIONS = 15`, the solver achieves
approximately `15/40 ≈ 37.5%` of perfect propagation per frame from one end to
the other. However, because the simulation runs at 120 Hz, corrections compound
rapidly over multiple frames, and the rope appears quite stiff.

### Why not a direct solver?

A tridiagonal direct solver (Thomas algorithm) could solve the chain constraints
in one pass. However:
- Iterative relaxation is simpler to implement
- It generalizes to non-chain topologies (meshes, cloth)
- With Verlet's position-based approach, iterative solving is the standard method
  (used in Hitman, GDC talks, etc.)

## 5. Fixed Timestep at 120 Hz

```rust
.insert_resource(Time::<Fixed>::from_hz(FIXED_HZ))
.add_systems(FixedUpdate, (verlet_integrate, apply_constraints).chain())
```

### Why fixed timestep for physics?

Verlet integration's accuracy depends on consistent `dt`. With variable
timesteps:
- Large `dt` causes instability (gravity overshoot, constraint violations)
- Small `dt` wastes computation
- The implicit velocity `x(t) - x(t-dt)` assumes equal-length timesteps

`FixedUpdate` runs at exactly `FIXED_HZ = 120.0` Hz regardless of the render
frame rate. If the game runs at 60 FPS, physics steps twice per frame. If it
runs at 240 FPS, physics steps every other frame.

### Why 120 Hz?

```
dt = 1/120 ≈ 0.00833 seconds
```

120 Hz is a common choice for physics simulations:
- Fast enough for stable rope simulation with 40 nodes
- Matches 120 Hz displays exactly (no interpolation artifacts)
- Divides evenly into common frame rates (60, 30)

Lower rates (e.g., 60 Hz) would require more constraint iterations for
equivalent stiffness. Higher rates (e.g., 240 Hz) would double the CPU cost
for diminishing returns.

### System chaining

```rust
(verlet_integrate, apply_constraints).chain()
```

The `.chain()` ensures Verlet integration runs before constraint solving every
fixed timestep. Without this, Bevy's scheduler might run them in any order or
in parallel, producing incorrect results (constraints must act on post-
integration positions).

## 6. Mouse Drag (Anchor Teleportation)

```rust
rope.positions[0] = world_pos;
rope.old_positions[0] = world_pos;
```

### Why set both positions?

Setting `old_positions[0] = world_pos` alongside `positions[0]` zeros the
implicit velocity of the anchor:

```
velocity = positions[0] - old_positions[0] = world_pos - world_pos = 0
```

If only `positions[0]` were set, the next Verlet step would compute a large
velocity from `positions[0] - old_positions[0]`, causing the anchor to "launch"
in the direction of the mouse movement. Zeroing implicit velocity makes the
anchor follow the mouse smoothly without momentum.

### Viewport-to-world conversion

```rust
let Ok(world_pos) = camera.viewport_to_world_2d(camera_transform, cursor_pos) else {
    return;
};
```

`viewport_to_world_2d` converts screen pixel coordinates (origin top-left,
Y-down) to world coordinates (origin center, Y-up). For a 2D camera with no
zoom or pan, this is:

```
world_x = cursor_x - window_width / 2
world_y = -(cursor_y - window_height / 2)
```

The Bevy method handles arbitrary camera transforms, including zoom and
translation.

## 7. Gizmo Rendering

```rust
fn draw_rope(rope: Res<Rope>, mut gizmos: Gizmos) {
    for i in 0..NODE_COUNT - 1 {
        gizmos.line_2d(rope.positions[i], rope.positions[i + 1], ROPE_COLOR);
    }
    for i in 1..NODE_COUNT {
        gizmos.circle_2d(rope.positions[i], NODE_RADIUS, NODE_COLOR);
    }
    gizmos.circle_2d(rope.positions[0], ANCHOR_RADIUS, ANCHOR_COLOR);
}
```

### Why Gizmos instead of Mesh2d?

Gizmos are immediate-mode: they are drawn for one frame and don't require
entity management. This is ideal for the rope because:
- The rope is a `Resource`, not a collection of entities
- Node positions change every frame
- No persistent mesh/material handles to manage
- Lines between nodes are trivial with `gizmos.line_2d`

The trade-off is that Gizmos are not batched as efficiently as mesh rendering,
but for 40 nodes this is negligible.

## Summary of Math

| Concept | Formula | Purpose |
|---|---|---|
| Verlet integration | `x_new = x + (x - x_old)*damping + a*dt^2` | Position-based dynamics with implicit velocity |
| Damping | `v *= 0.99` per frame | Energy dissipation for settling |
| Distance constraint | `correction = (B-A) * 0.5 * (d-L)/d` | Maintain fixed segment length |
| Iterative relaxation | 15 Gauss-Seidel passes | Converge toward global constraint satisfaction |
| Fixed timestep | 120 Hz via `FixedUpdate` | Deterministic, stable physics |
| Anchor teleport | Set both `pos` and `old_pos` | Zero implicit velocity on drag |

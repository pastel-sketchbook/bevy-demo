# Pong -- Math Rationale

## Overview

The pong demo implements a classic two-paddle game with a twist: the ball
follows quadratic Bézier curves instead of straight lines. The math covers
Bézier curve evaluation, duration-from-distance speed control, AI paddle
tracking with capped speed, and AABB collision detection for paddle hits.

## 1. Quadratic Bézier Curve

```rust
fn evaluate(&self, t: f32) -> Vec2 {
    let t = t.clamp(0.0, 1.0);
    let inv_t = 1.0 - t;
    // Quadratic bezier: B(t) = (1-t)^2*P0 + 2(1-t)tP1 + t^2*P2
    self.start * (inv_t * inv_t) + self.control * (2.0 * inv_t * t) + self.end * (t * t)
}
```

### Definition

A quadratic Bézier curve is defined by three points:
- **P0** (start): where the curve begins
- **P1** (control): influences curvature but is not on the curve
- **P2** (end): where the curve ends

The parametric form with parameter `t ∈ [0, 1]`:

```
B(t) = (1-t)^2 * P0 + 2(1-t)t * P1 + t^2 * P2
```

### Basis functions (Bernstein polynomials)

The three coefficients are the degree-2 Bernstein basis polynomials:

```
B_0(t) = (1-t)^2      -- weight on P0, starts at 1, ends at 0
B_1(t) = 2(1-t)t      -- weight on P1, peaks at t=0.5
B_2(t) = t^2           -- weight on P2, starts at 0, ends at 1
```

They sum to 1 for all `t`, guaranteeing the curve stays within the convex hull
of the three control points.

### Properties

- `B(0) = P0` (starts at start point)
- `B(1) = P2` (ends at end point)
- The tangent at t=0 points from P0 toward P1
- The tangent at t=1 points from P1 toward P2
- The curve is smooth (C-infinity) and has no cusps

### Why Bézier for Pong?

Traditional Pong uses straight-line ball trajectories. Bézier curves add visual
interest: the ball arcs through the playing field, making it harder for the AI
to predict. The control point's vertical offset creates banana-shaped paths that
curve around paddles.

## 2. Control Point Construction

```rust
let mid_x = (start.x + end_x) / 2.0;
let curve_strength = rng.random_range(-half_h * 0.6..half_h * 0.6);
let control = Vec2::new(mid_x, start.y + curve_strength);
```

### Geometry

The control point is placed at the horizontal midpoint between start and end,
with a random vertical offset from the start Y position. This creates symmetric
horizontal progression with asymmetric vertical curvature.

```
control.x = (start.x + end.x) / 2    -- horizontal midpoint
control.y = start.y + random offset   -- vertical perturbation
```

### Curvature strength

The `curve_strength` is sampled from `[-0.6 * half_h, 0.6 * half_h]`. With
`WINDOW_HEIGHT = 1024`, `half_h = 512`, so:

```
curve_strength ∈ [-307, 307] pixels
```

This means the control point can be up to 307 pixels above or below the start
point, producing anything from nearly straight paths (small offset) to extreme
arcs (large offset).

### Maximum curvature

For a quadratic Bézier, the maximum deviation from the straight line P0→P2
occurs at t=0.5 and equals `0.5 * distance(midpoint_of_P0P2, P1)`. So the
ball can deviate up to about 154 pixels from a straight trajectory.

## 3. Duration from Distance

```rust
let distance = start.distance(end);
let speed = rng.random_range(MIN_BALL_SPEED..MAX_BALL_SPEED);
let duration = distance / speed;
```

### The math

```
duration = |P2 - P0| / speed
```

This computes the time to traverse the path based on the **chord length**
(straight-line distance from start to end), not the **arc length** (actual
distance along the curve). The arc length of a quadratic Bézier has no
closed-form solution (it requires an elliptic integral), so the chord length
is a practical approximation.

### Consequence

Because the arc is always longer than the chord (except for a straight line),
the ball's visual speed along the curve is slightly faster than the nominal
`speed` value. For moderately curved paths, the difference is small. For extreme
curves, the ball appears to accelerate through the arc -- which actually adds to
the gameplay feel.

### Speed range

```
MIN_BALL_SPEED = 200.0
MAX_BALL_SPEED = 500.0
```

For a cross-screen path (~1600px chord), the duration ranges from:

```
slow: 1600 / 200 = 8.0 seconds
fast: 1600 / 500 = 3.2 seconds
```

### Parameter advancement

```rust
path.t += time.delta_secs() / path.duration;
```

The parameter `t` advances uniformly in time: `dt_param = dt_real / duration`.
After `duration` seconds, `t` reaches 1.0 and the ball arrives at the end point.
This provides constant parametric speed, though not constant spatial speed (the
ball moves faster where the curve is more "stretched").

## 4. AI Paddle Tracking

```rust
let diff = ball_y - transform.translation.y;
let move_amount = diff.signum() * ai_speed.min(diff.abs()) * time.delta_secs();
transform.translation.y += move_amount;
transform.translation.y = transform.translation.y.clamp(-max_y, max_y);
```

### Decomposition

1. **`diff`**: signed distance from paddle center to ball Y
2. **`diff.signum()`**: direction to move (+1 up, -1 down)
3. **`ai_speed.min(diff.abs())`**: speed is capped at `AI_REACTION_SPEED * PADDLE_SPEED`,
   but also capped at the remaining distance to prevent overshooting
4. **Multiply by `dt`**: frame-rate-independent movement

### Why `min(diff.abs())`?

Without this, when the paddle is very close to the ball (say 2px away), it would
overshoot by `ai_speed * dt - 2` pixels, causing oscillation around the target.
The `min` ensures the paddle never moves farther than the remaining gap,
providing smooth arrival without overshoot.

### AI imperfection

```rust
const AI_REACTION_SPEED: f32 = 0.85;
let ai_speed = PADDLE_SPEED * AI_REACTION_SPEED;
```

The AI moves at 85% of maximum paddle speed. This deliberate handicap means the
ball can sometimes outrun the paddle on extreme curves, preventing the AI from
being unbeatable. Since `PADDLE_SPEED = 400` and max ball speed is 500, a ball
moving vertically at its speed limit will outpace the AI.

## 5. AABB Paddle Collision

```rust
// Left paddle collision
ball_transform.translation.x - ball_r < paddle.translation.x + paddle_half_w
    && ball_transform.translation.x > paddle.translation.x
    && ball_transform.translation.y < paddle.translation.y + paddle_half_h
    && ball_transform.translation.y > paddle.translation.y - paddle_half_h
```

### AABB (Axis-Aligned Bounding Box) test

This checks if the ball's center (offset by radius on the x-axis) overlaps
the paddle's rectangle:

1. **Left edge**: ball's left side (`x - r`) is past paddle's right edge
   (`paddle_x + half_w`) -- ball has entered the paddle's x-range
2. **Right guard**: ball center is still right of paddle center -- prevents
   triggering after the ball has passed through
3. **Vertical bounds**: ball center is within paddle's vertical extent

### Why not circle-rectangle intersection?

A proper circle-rectangle test would check the distance from the ball center to
the nearest point on the rectangle. The simplified AABB test treats the ball as
a point offset by its radius on the x-axis only. This is adequate for Pong
because:
- The ball moves predominantly horizontally
- Vertical misses are handled by the y-range check
- Corner cases (ball hitting paddle corner at an angle) are rare and
  non-critical for gameplay

### Collision response

```rust
ball_transform.translation.x = paddle.translation.x + paddle_half_w + ball_r;
let current_pos = Vec2::new(ball_transform.translation.x, ball_transform.translation.y);
*path = BezierPath::new_path(current_pos, true, &mut rng.0);
```

On collision:
1. Snap ball to paddle's right edge (prevent embedding)
2. Generate a new Bézier path going in the opposite direction

This replaces the traditional velocity-reflection model with a path-regeneration
model. Instead of `v.x = -v.x`, the ball gets an entirely new curved trajectory.

## 6. Scoring and Ball Reset

```rust
let scored = if ball_transform.translation.x < -half_width {
    score.right += 1;
    true
} else if ball_transform.translation.x > half_width {
    score.left += 1;
    true
} else {
    false
};
```

### Scoring detection

The ball has "scored" when its x-position exceeds the window boundary. The end
point of each Bézier path is placed past the edge (`half_w + BALL_RADIUS * 2`),
so a ball that isn't intercepted by a paddle will naturally reach the scoring
zone.

### Ball randomization on score

```rust
let new_radius = rng.0.random_range(8.0..20.0);
radius.0 = new_radius;
mesh.0 = meshes.add(Circle::new(new_radius));

let r = rng.0.random_range(0.5..1.0);
let g = rng.0.random_range(0.5..1.0);
let b = rng.0.random_range(0.5..1.0);
material.0 = materials.add(Color::srgb(r, g, b));
```

On each score, the ball gets a new random radius (8-20px) and random bright
color. The mesh and material are recreated via `meshes.add()` and
`materials.add()`, which returns new `Handle` values. The old assets are not
explicitly freed -- Bevy's asset system will garbage-collect unreferenced assets.

### Rust idiom: `is_changed()` guard

```rust
if score.is_changed()
    && let Ok(mut text) = query.single_mut()
{
    *text = Text2d::new(format!("{} - {}", score.left, score.right));
}
```

`is_changed()` is a Bevy change-detection method on `Res<T>`. It returns `true`
only in frames where the resource was mutated. This avoids reformatting the
score string every frame when nothing changed -- a minor optimization but
idiomatic Bevy practice.

## Summary of Math

| Concept | Formula | Purpose |
|---|---|---|
| Quadratic Bézier | `B(t) = (1-t)^2*P0 + 2(1-t)t*P1 + t^2*P2` | Curved ball trajectory |
| Control point | `(midpoint_x, start_y + random_offset)` | Curvature variety |
| Duration from chord | `duration = chord_length / speed` | Approximate travel time |
| Parametric advance | `t += dt / duration` | Constant-time traversal |
| AI tracking | `move = signum(diff) * min(speed, |diff|) * dt` | Smooth chase with overshoot prevention |
| AABB collision | 4-condition axis-aligned overlap test | Paddle hit detection |
| Collision response | Regenerate Bézier in opposite direction | Path-based reflection |
